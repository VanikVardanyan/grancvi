# Master Onboarding Redesign — Design

## Контекст и проблема

Текущий self-service флоу мастера (с лендинга `grancvi.am` через CTA «Стать мастером» → `t.me/grancviWebBot?start=signup`):

1. Бот шлёт сообщение с кнопкой `Записаться` (RU дефолт; «booking»-копирайт, не «регистрация»).
2. TMA: `RegisterSelf` — имя + slug + multi-select специализаций.
3. POST `/v1/register/master/self` → создаётся `Master` с `is_public=false` (модерация).
4. TMA редирект на `Onboarding`-страницу: чек-лист «часы работы» + «услуги».
5. После обоих шагов или «Пропустить» → дашборд мастера.
6. Публичная страница `grancvi.am/{slug}` 404 пока админ не одобрил.

**Проблемы:**

- Кнопка «Записаться» — клиентская копирка, смыслово неверна для signup-пути.
- Дефолт языка определяется по `language_code` Telegram → армянские пользователи с английским/русским Telegram получают русский интерфейс. Противоречит Armenia-first продукту.
- Модерация ломает основной wedge GTM («физический QR работает за 3 секунды»). Между регистрацией и одобрением QR-стикер бесполезен.
- Чек-лист `Onboarding` ощущается как «домашка»: услуги — отдельная страница, часы — псевдо-шаг (дефолт уже стоит).
- Нет момента гордости после регистрации — мастер не видит свою страницу/QR.
- «Услуги» как отдельный шаг частично дублируют «специализацию», только что выбранную.

## Цели

1. **Один экран регистрации:** имя + slug + специализации + услуги (предвыбраны) + сабмит → дашборд.
2. **Welcome-экран сразу после регистрации:** имя/специальность как заголовок, QR с публичной ссылкой, кнопка «Скопировать ссылку» — момент гордости, точка для шеринга.
3. **Без модерации на пути:** мастер сразу `is_public=true`, страница работает, QR scannable. Защита от squat'инга — reserved-slug list. Защита от абьюза — post-hoc `/block`.
4. **Корректный копирайт лаунчера** под контекст deep-link'а.
5. **Армянский по умолчанию.**

## Out of scope

- Salon self-service (`RegisterSelfSalon`) — те же паттерны применимы, но другие данные/UI; отдельная задача.
- Редизайн редактора расписания/услуг (используем существующие страницы как «edit later»).
- Каталог / поиск.
- Удаление admin moderation UI — `/admin` остаётся, просто становится менее критичным (используется для `/block`/`/unblock`, не для approve).

## Детали UX — RegisterSelf

### Раскладка (вертикальный скролл)

```
┌─ Имя ───────────────────────┐
│ [____________________]       │
└─────────────────────────────┘

┌─ Ссылка ────────────────────┐
│ grancvi.am/[anna__________]  │
│ нельзя менять чаще раз в …   │
└─────────────────────────────┘

Чем занимаешься?
[● Колорист] [● Парикмахер ж.] [○ Маникюр]
[○ Брови] [○ Косметология] [○ Ещё]

──────────────────────────────
Услуги — 7 выбрано

▼ Колорист                4 / 4
   ☑ Однотонное окрашивание · 120м
   ☑ Мелирование · 150м
   ☑ Балаяж · 180м
   ☑ Окрашивание корней · 60м

▶ Парикмахер ж.           3 / 3

   [+ Своя услуга]

──────────────────────────────
TG MainButton:  Готов начинать (7 услуг)
```

### Поведение

- **Имя**: обязательное.
- **Slug**: автогенерация транслитом по имени (как сейчас в `RegisterSelf.tsx`). Можно править. Проверка через существующий `SlugInput` (debounced async-валидация).
- **Специализации**: чипы, multi-select. При тапе чипа — сразу появляется/исчезает соответствующая аккордеон-карточка ниже. Лимит — soft, без enforcement (предупреждение если >3, но всё ещё сабмитится).
- **Аккордеон по специализации**:
  - Заголовок: название + counter «4 / 4».
  - **Default**: при первом выборе специализации её аккордеон сразу раскрыт. При выборе следующей — её аккордеон **свёрнут** (первая остаётся раскрытой, чтобы не сбивать контекст). Пользователь раскрывает руками если хочет refine.
  - При повторном тапе чипа специализация удаляется (карточка пропадает, услуги де-выбираются).
- **Услуги внутри аккордеона**: чекбоксы с дефолтом ✓. Имя + длительность. Цена не показывается (дефолт `null`, мастер ставит позже в редакторе).
- **«+ Своя услуга»** — кнопка под всеми аккордеонами. Открывает существующую `ServiceForm` (используем как есть, не вырезаем). Добавленные кастомные услуги показываются отдельной строкой над «+».
- **TG MainButton (sticky bottom)**: текст «Готов начинать · {N} услуг», `enabled` если имя ≥ 1 char + ≥ 1 специализация + ≥ 1 услуга. Loading state во время POST.
- **Browser fallback** (когда не TMA): inline-кнопка под формой как сейчас.

### Submit

- POST `/v1/register/master/self` с `{name, slug, specialty: codes.join(','), lang}`.
- После успеха: серия POST на `/v1/master/services` (как в `PresetPicker` сейчас) для каждой выбранной услуги: `{name, duration_min, preset_code, price_amd: null}`.
- POST `/onboarded` — сразу же отмечает мастера как онбордингованного (никакой второй стадии).
- Навигация: `/welcome`.

### Ошибки

- `slug_taken` / `slug_invalid` / `slug_reserved` — inline под полем slug, фокус на slug.
- `already_registered` — toast + редирект на `/`.
- Network — сохраняем введённое в localStorage, `Retry` кнопка.

### Reserved slugs

В `src/services/slugs.py` добавить:

```python

    # бренды + системные
    "grancvi", "admin", "support", "help", "info", "api", "app",
```

**Финальный список reserved-slugs утверждается до коммита кода**, не в спеке — он чисто content-decision. Я подготовлю топ-50 имён + список бренд-слов и пришлю на одобрение перед мерджем.

Список держится в коде (не в БД) — простой ребилд, миграции не нужны. Отдельный admin-эндпоинт для редактирования — out of scope первой итерации.

## Координация деплоя

Бэк и фронт меняются вместе. Стратегия — **обратная совместимость в коде**:

1. **Бэк деплоится первым:**
   - `register_self` начинает создавать `is_public=true`.
   - Reserved-slug проверка активна.
   - Новый лаунчер-копирайт работает.
   - Старый фронт продолжает дёргать `/v1/register/master/self` и редиректить на `/onboarding`. Это всё ещё работает — `Onboarding.tsx` существует, чек-лист отображается, но первый шаг (часы) уже галочкой ✓ из-за дефолта, второй (услуги) — стандартный флоу. Не сломано.

2. **Фронт деплоится вторым:**
   - Новый `RegisterSelf.tsx` собирает услуги inline и шлёт их сразу.
   - Редирект с `/register/self` идёт на `/welcome`, не `/onboarding`.
   - `Onboarding.tsx` удаляется + роут.

После полного деплоя — старый `Onboarding`-флоу больше не доступен; пути на него не ведут. Если кто-то с открытой старой вкладкой попробует — получит 404, обновит страницу, попадёт на новый.

## Риски

1. **Slug-squatting**: reserved-list ловит самые ценные имена, но не уникальные («ани»/«мария»/etc.). Митигация: list расширяется по мере появления абьюзов; `/block` доступен админу.
2. **Тестировани на проде**: пользователь сознательно отказался от стенда. Минимизируем риск через:
   - Поэтапный деплой (бэк → фронт)
   - Мониторинг ошибок через PostHog/structlog первые 24ч
   - Возможность быстрого rollback одним PR-revert
3. **QR на Welcome ломается на медленных устройствах**: генерация PNG в браузере дёшевая, но всё равно может тормозить. Размер фиксируем 256×256 (плотнее ECC), генерация ~50ms на iPhone 8 — приемлемо.
4. **Welcome-страница на медленном API**: пока не подгружены services/profile, показываем skeleton.

## Метрики успеха

- Доля self-service регистраций, доходящих до `onboarded_at != null` за **одну сессию**: цель ≥80% (сейчас ≤50% из-за двух-этапного флоу — оценочно).
- Время от `/start signup` до первой созданной услуги: цель ≤2 мин (сейчас ≥5).
- Доля мастеров, поделившихся ссылкой в первые 24ч после регистрации (если можем трекать): baseline.
- Число `/block`-ов в неделю: стабилизироваться ниже 1/нед (показатель что reserved-list достаточен).

## Открытые вопросы (отложены, не блокируют)

- Salon self-service — те же паттерны, но требуют отдельной проработки (нет специализаций → нет аккордеона; вместо этого — ввод адреса/часов).
- Многоязычный QR — сейчас QR ведёт на `grancvi.am/{slug}`, тот сам определяет язык клиента. Если нужно жёстко закрепить — добавим `?lang=hy` в QR. На первую итерацию не делаем.
- Reserved-list админ-API — пока хардкод. Если spam начнётся — выносим в БД с эндпоинтами.
